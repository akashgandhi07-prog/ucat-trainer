# Question Storage Summary for LLMs

This project has 15 public UCAT skills trainers, listed in `src/data/landingTrainers.ts`.
Their "questions" are stored in three different ways:

- Static TypeScript banks in `src/data/**`
- Supabase tables seeded from JSON or generated rows
- Procedural generators that create questions at runtime from templates/rules

An LLM improving the question bank should first identify which storage pattern a trainer uses. Do not assume every trainer has a single editable JSON file.

## Runtime Patterns

### Verbal Reasoning

`PASSAGES` in `src/data/passages.ts` is the core content bank for Speed Reading, Rapid Recall, Keyword Scanning, and Inference.

Speed Reading and Rapid Recall use `DistortionQuiz` to build True/False/Can't Tell style questions dynamically from the selected passage. Keyword Scanning selects target words dynamically from the passage. Inference is different: it has hand-authored inference prompts and exact evidence spans in `src/data/inferenceQuestions.ts`.

### Decision Making

Syllogism trainers are Supabase-backed. Their questions live in `public.syllogism_questions`, with `question_mode` separating `foundation`, `micro`, and `macro`. Foundation rows are in migrations; micro/macro rows are generated by `src/utils/SyllogismGenerator.ts` and seeded with `scripts/seedSyllogisms.ts`.

The three non-syllogism DM trainers use `public.dm_trainer_questions` through the `get_dm_trainer_drill` RPC. Their editable local source is under `src/data/dmTrainers/`, and `npm run seed:dm-trainers` exports those TypeScript questions to JSON and upserts them into Supabase.

### Quantitative Reasoning

Calculator and Mental Maths are mostly procedural. They do not have one stored bank of fixed questions. To improve them, edit their generators and validation scripts.

Conversions is the exception: it is a static local bank in `src/data/conversionQuestions.ts`.

### Situational Judgement

SJT questions are authored in `scripts/sjtQuestions.source.ts`, exported to `supabase/seed/sjt_questions.json`, then upserted into `public.sjt_questions`. The frontend retrieves one random question at a time via `get_random_sjt_question`. Direct client reads are intentionally blocked by RLS.

## Recommendation for Future Question Expansion

For an LLM-assisted question expansion workflow, keep these lanes separate:

1. Passage-based VR: add or refine passages, then test generated questions for quality.
2. Authored inference: add explicit question text, exact evidence span substrings, and explanations.
3. DM local banks: add full MCQ objects with explanations, traps, and review metadata, then export/seed.
4. SJT: add structured scenarios to `scripts/sjtQuestions.source.ts`, export JSON, then seed.
5. Procedural QR/syllogism: improve templates/generators and run verifier scripts, rather than writing isolated one-off questions.
